### 1e (desktop slice) — `dx build --platform desktop` actually works now (2026-07-15)

The user asked for this explicitly ("don't go crazy") after 1e's web half — scoped tight: stand up the desktop build and get real filesystem storage working, nothing more (no vault-path config UI, no auth persistence on desktop, no Windows/Linux-specific testing beyond what the `directories` crate already handles generically).

**It didn't just work by flipping `--platform desktop` — three real, verified-not-assumed problems, each fixed at the root cause rather than patched per call site:**

1. **`dx build --platform desktop` compiles clean but the binary panics on launch.** Root cause: `default = ["web"]` in `Cargo.toml` stays active even for a desktop build (`dx` doesn't pass `--no-default-features`), so every unconditional `web_sys::window()` call (main.rs's startup effect, navbar.rs's session check + token-refresh loop, moment.rs's sort_mode persistence, auth.rs's login/logout) tries to run on a native (non-wasm) target and panics: `cannot access imported statics on non-wasm targets`. Existing code already had one `#[cfg(feature = "web")]` guard in main.rs, but it didn't work for the same reason — `web` being *present* isn't the right signal when it's always present. Fixed by gating every one of those call sites on `#[cfg(not(feature = "desktop"))]` instead (presence of `desktop`, not absence of `web`) — desktop just has no session/preference persistence yet, a known, deliberately-deferred gap, not solved here.
2. **Same failure mode, but for the `clog!` debug-logging macro** (`web_sys::console::log_1`, used at dozens of call sites app-wide) — fixed once at the macro definition in `main.rs` rather than touching every call site, since gating each one individually would have been exactly the kind of scope creep the "don't go crazy" ask was warning against.
3. **`LocalStorage` itself needed a second, real implementation** — new `src/api/local_desktop.rs`, plain `std::fs` + the `directories` crate (added as a desktop-only optional dependency) resolving `~/Documents/Peeplist/` the same way on every OS. Same shape as the web backend: one file per entity under `people/`, `self.md` at the vault root, atomic writes (`.md.tmp` + rename, per §1d), hard deletes (no `trash.yaml` yet, matching the web backend's same deliberate scope cut), lookups by parsing each file's `id:` field rather than trusting the filename (per §1d's own "filename is a creation-time hint, not the source of truth" rule). `storage.rs` picks between this and the web `LocalStorage` via `#[cfg(feature = "desktop")]` on the `pub use`, so `ActiveStorage`'s code doesn't change at all.

**Verified, not assumed** (per the plan's own instruction to confirm rather than guess about the desktop webview): `dx build --platform desktop` compiles clean, the binary boots and stays running (previously panicked within milliseconds), and — best evidence available — real files appeared under `~/Documents/Peeplist/` (`self.md` auto-created on first launch, plus two people created through the actual live window, including the entity-type dropdown correctly offering `Friend`/`Family`/etc. and persisting on disk with correct YAML). Both `cargo check` (wasm32 + native) and the full `dx serve` web smoke pass (including a fresh local-vault create/reload round-trip) were re-run after all of this and are unaffected — this phase only *added* gating, it didn't change any web-path behavior.

**Not verified this session, worth checking next time the app is opened on desktop**: whether Graph View's d3-force JS interop (`document::eval`, `src/components/graph.rs`) actually renders correctly inside the desktop webview — this environment couldn't get a real screenshot of the native window (no attached display for automated screenshot tools), so this is the one item from the plan's own flagged-risk list that's still just "probably fine" rather than confirmed.

### 1e (web slice) — the Local vault is real now (2026-07-15)

New `src/api/local.rs`: `LocalStorage` is no longer a stub. One `localStorage` key per entity (`peeplist_vault:entity:<id>`), holding exactly what `vault_format::render_entity_file` produces — the same text a desktop build will eventually write to disk, so no format translation to get wrong later. Full CRUD implemented: reads scan all `peeplist_vault:entity:*` keys (no separate index to keep in sync); `get_moments()` flattens every entity's embedded moments; field-patches (`update_moment_field`/`update_entity_field`) do the generic JSON-merge-patch the plan describes (load record → `serde_json::Value` → merge the one changed field → re-deserialize → re-render → write back).

**A real footgun caught here, not just theoretical**: `MomentType.created_at`/`EntityType.created_at` are `#[serde(skip_serializing)]` — sensible for the Supabase PATCH-body use case (never send a client timestamp back over a server-assigned one) but that also means a plain `serde_json::to_value(&record)` silently drops `created_at`, and it would've come back as `""` after every single local field-patch (`default` kicks in on deserialize). Fixed by explicitly re-inserting `created_at` into the `Value` before merging — see `to_patchable_value` in `local.rs`.

**Deliberately deferred, not forgotten**: no `.peeplist/trash.yaml` wiring yet — deletes are hard deletes for now. §1d's own reasoning ("non-destructive, no recovery UI needed yet") means this is invisible to the user either way today; revisit only if a recovery UI is ever actually wanted. `get_entity_types()` seeds a small hardcoded default list (`Friend`/`Family`/`Colleague`/`Partner`) plus whatever type names are actually in use — not meant to be exhaustive, just non-empty on a brand new vault.

**`active_vault`'s default was deliberately left as `Synced`, not flipped to `Local`** despite Local now being real — flipping it would suddenly empty the view for anyone currently using the Synced vault (this session's actual usage) with no warning. That flip is a separate, explicit decision to make once the user's ready for it, not something to bundle in silently. Local is fully reachable today via the vault switcher's "Local" option (or automatically once logged out, since `Synced` + no token already falls back to `Local`).

**A second real bug, and the fix that closes the whole class of it**: `active_vault`'s raw signal value can say `Synced` even when there's no token (that's the deliberate default above) — anything that read the raw signal directly for a *display or logic* decision (not just to construct `ActiveStorage`) would disagree with what the storage layer actually does. This had already caused the vault-switcher checkmark bug fixed the day before, and caused a second one here: `MomentInputCmp` was defaulting new moments with no entity selected to the Supabase self id (`"0"`) even while actually writing to the Local vault, because it read `active_vault` raw. Fixed properly this time instead of patching the one call site: added `VaultKind::effective(self, token: &Option<String>) -> VaultKind` (mirrors `ActiveStorage::for_vault`'s own fallback exactly) and routed every UI spot that needs "what's actually active" through it, including refactoring the switcher's existing inline version of the same logic to use the shared method.

**Self-entity id also needed a real per-vault resolution, not just a note that it was deferred**: added `VaultKind::self_entity_id(&self)` (`"0"` for Synced, `vault_format::LOCAL_SELF_ENTITY_ID` for Local) and `api::is_self_entity(id)` (checks against both sentinels at once, for spots that just need "is this the self entity" without needing to know which vault). `MomentInputCmp` now resolves the correct self id per vault instead of hardcoding the Supabase one.

**Found and fixed while actually looking at real Local-vault data for the first time** (previously invisible since Local never had real data to expose the bug): the self entity was showing up as a normal, clickable entity in the sidebar's entity list, and in the moment-composer's entity-picker dropdown (redundantly, since it's already the default option there), and skewing `ab_stats_cmp`'s "most active" ranking. All three fixed with the same `is_self_entity` filter, same pattern already applied in Graph View. This was a known, explicitly-flagged-as-unaudited gap (see memory `project_self_entity_convention`) that just never had real data to surface it before now.

Verified: `cargo check` (wasm32 + native) clean, all 7 `vault_format` tests still pass, and a full `dx serve` + Playwright round-trip — create a person and a task while on the Local vault, confirm the rendered YAML in `localStorage` looks right, **reload the page, confirm both persist** (the actual point of this phase), confirm Graph View and the sidebar entity list correctly show only real entities, not the self entity.

## Taskwarrior-style attributes, part 1: depends-on enforcement (2026-07-15)

The user's actual ask is bigger than this — "everything taskwarrior has" (`project:`, `priority:`, `due:`, `scheduled:`, `until:`, `recur:`, `+tag`), buried in a collapsible "advanced" fold on the moment detail panel so it doesn't clutter the list view, all feeding into a smarter urgency formula. Scoped down deliberately after discussion: `recur` (recurring tasks) is deferred to its own pass — it's a real design question (does completing a recurring task spawn the next instance immediately? on what schedule? what happens to the finished one?), not just a new field. Multi-dependency (taskwarrior allows depending on several tasks) is also deferred — today's single `depends_on` stays single, no schema migration needed. The urgency-formula rewire and the `key:value` quick-capture text parsing are explicitly "come back to later" per the user. **This pass was just the one piece the user flagged as broken first: depends-on didn't actually do anything.**

**The bug**: `depends_on` existed structurally (`MomentType.depends_on`, a "Depends on" picker in `ab_task_cmp`, `compute_urgency`'s blocked/blocking scoring) but nothing actually *enforced* it — you could complete a moment that depended on an incomplete one. Purely informational, not a real constraint.

**Fixed**: `CheckboxProps` gained a `disabled` field (`src/types.rs`); `CheckboxCmp` renders it as visually disabled (`src/components/moment.rs`) and now also refuses the `on_change` call itself as a second guard, not just relying on the HTML `disabled` attribute. Wired into all three places a moment can be marked complete: `MomentCmp`'s list-row checkbox (which now also computes its own `is_blocked` from `state.moments` and shows a small red "Blocked" label next to the title, same visual language as the existing "Promise" label), `ab_task_cmp`'s detail-panel checkbox, and its "Complete with reaction" button.

**A second, real bug found while wiring this up, not just "add the guard"**: `ab_task_cmp`'s `is_blocked`/dependency-dropdown-selected-state was computed from `moment.depends_on` — `moment` being a one-time snapshot taken from `current_moment` when the panel opens. The dependency `<select>`'s own `oninput` handler updates the global `moments` signal (so the list view and `dependency_candidates` already reflected changes live) but never updates `current_moment`, so **within the same open panel session**, picking a dependency didn't actually disable the checkbox (or show as selected in the dropdown) until you closed and reopened the panel — verified via Playwright before assuming the fix worked. Fixed by reading `depends_on` live off the `moments` signal by id (`current_depends_on`) instead of off the stale snapshot, used consistently for `is_blocked`, the dropdown's `selected` state, and the "Blocked by" message.

Verified end-to-end via `dx serve` + Playwright: created two tasks, set one to depend on the other, confirmed both the list-row and detail-panel checkboxes disabled with no page reload, confirmed clicking the disabled checkbox is a no-op, completed the dependency, confirmed the dependent task's checkbox re-enabled and the "Blocked" label disappeared — all reactively, live. `cargo check` (wasm32 + native) and the full test suite unaffected.

**Still outstanding after part 1**: `priority`, `project`, `scheduled`, `until` fields; the advanced-fold UI to hold them; wiring any of this into `compute_urgency`; `recur`; quick-capture text parsing. See "part 2" below for the fields + fold — `compute_urgency`, `recur`, and quick-capture parsing remain unstarted.

### Follow-up, same day: checkbox staleness + cascading un-completion

Two more real bugs, reported by the user immediately after using the depends-on enforcement above.

**Bug 1 — the detail panel's checkbox went stale when switching between moments without closing the panel.** Checking a task complete, then clicking a *different* task in the list (panel stays open, only `current_moment` changes), showed the new task's checkbox as still checked — required an uncheck-then-recheck to register. Root cause turned out to be a genuine DOM quirk, not a data bug (confirmed: `moment.title` updated correctly on switch, proving the component *was* re-rendering with fresh data; `localStorage` had the right value the whole time) — once a `<input type="checkbox">`'s live DOM `checked` *property* has been touched by a real click, simply changing the declared value on the same persisted DOM node doesn't reliably re-sync it, and this panel reuses the same DOM subtree across different moments (it only remounts on open/close, not on moment-switch). A `key` on the single always-present `CheckboxCmp` child did **not** fix it — confirmed empirically that Dioxus only honors `key` for reconciling siblings inside a `for`, not a lone child at a fixed position. Fixed with the same single-iteration `for _key in [id.clone()] { CheckboxCmp { key: "{_key}", ... } }` trick already proven elsewhere in this file (`MomentListCmp`'s drag rows), which forces a genuinely fresh DOM node per moment id.

**Bug 2, actually a product decision the user made explicitly**: completing X, completing Y (which depends on X), then un-completing X left Y sitting completed — blocked-and-disabled again, but still flagged "done." The user's call: don't allow that state to exist at all — "have the user untangle the blocked-blocking dependency chain," i.e. un-completing a moment must cascade to un-complete everything that transitively depends on it. New `cascade_uncomplete()` in `moment.rs`: walks the dependency graph breadth-first from the un-completed moment, un-completing (and persisting) every completed dependent, transitively. Wired into both places a moment gets un-completed (`MomentCmp`'s row checkbox, `ab_task_cmp`'s detail checkbox — the "Complete with reaction" path only ever completes, never un-completes, so it doesn't need this).

**A third bug found verifying the cascade, same root cause as Bug 1's class but different mechanism**: `MomentCmp`'s row checkbox used a `use_signal` (`is_completed`) seeded once at mount from `props.moment.completed_at` — correct for the row's *own* direct clicks, but when the cascade un-completes a *different* row than the one clicked, that row's own signal never gets touched and stays stale. Fixed by dropping the separate signal entirely and deriving `checked` straight from `props.moment.completed_at` on every render — matching the simpler pattern `ab_task_cmp`'s checkbox already used correctly. (The `visual_opacity` fade-on-complete animation keeps its own local signal — that's genuinely ephemeral UI state, not a data-truth question, so it was left alone.)

Verified end-to-end via `dx serve` + Playwright, all without any page reload: switching between two open moments no longer shows a stale checkbox; completing X then Y then un-completing X correctly un-completes Y too, both in the persisted YAML and in the live checkbox/disabled/"Blocked"-label state of a row that was never itself clicked. `cargo check` (wasm32 + native) and the full test suite unaffected.

## Taskwarrior-style attributes, part 3: quick-capture text parsing + @-mention entity chooser (2026-07-15)

The input-method piece explicitly deferred in part 2. New pure module `src/quick_capture.rs`: `tokenize()`/`parse()` turn a raw title string plus the current entity list into a `ParsedCapture` (title with recognized syntax stripped out, plus `entity_id`/`priority`/`project`/`due_at`/`scheduled_at`/`until_at`/`tags_add`). Recognized syntax: `priority:H/M/L` (also `high`/`medium`/`low`), `project:<value>`, `due:`/`scheduled:`/`until:` (accepts `today`, `tomorrow`, `YYYY-MM-DD`, or `YYYY-MM-DDTHH:MM`), `+tag`/`-tag`, and `@<entity name>`. Anything unrecognized — including a malformed date or a `@name` that doesn't match a real entity — is left as plain text and folds back into the title; that's the "didn't activate" signal by design, not a parse error. 8 unit tests cover extraction, longest-entity-name-wins matching, unmatched mentions, invalid dates, and a token-reconstruction invariant (tokens rebuild the exact original string — required for the overlay below to render without gaps).

**@-mention detection is end-of-string, not cursor-position-aware**: `trailing_mention_query()` just checks whether the input's last whitespace-delimited word starts with `@`. Deliberate — quick capture is typed left-to-right at the end of the field, and this sidesteps needing `web_sys` DOM selection APIs, which would have re-opened the same web-vs-desktop split already worked around for `window()`/`clog!` elsewhere (see part 1e). `apply_mention()` replaces just that trailing fragment with the full matched entity name plus a trailing space, so once selected, the mention becomes exact and unambiguous for `tokenize()` from then on.

**UI**: new `QuickCaptureInput` component (`src/components/moment.rs`) replaces the plain title `Input` in `MomentInputCmp`. It's a real `<input>` (owns focus/typing/caret, but its own text is set fully transparent) with a decorative `<div>` absolutely positioned on top (`pointer-events: none`) rendering the same tokens as colored `<span>`s — recognized tokens in the theme's `HL` color and bold, everything else in normal text color. This "invisible input + colored backdrop" stacking is the standard lightweight way to fake syntax highlighting in a plain text field without a contenteditable-based editor. An @-mention dropdown (filtered, longest-prefix-sorted, capped at 8, arrow-key/Enter/Tab/click to select) appears under the field whenever `trailing_mention_query()` is `Some`.

**Wiring into submit**: `MomentInputCmp`'s `submit_moment` now parses the raw title through `quick_capture::parse()` before building `NewMomentType` — `parsed.entity_id` (from an `@mention`) overrides the entity dropdown selection when present; after creation, if any advanced fields were parsed, they're PATCHed onto the new moment's `metadata` (and `due_at` separately, since it's a top-level `MomentType` field, not part of the jsonb blob) using the same pattern as `patch_moment_metadata` from part 2.

### Follow-up, same day: quoted `@"..."` / `+"..."` / `-"..."` for names and tags with spaces

The bare (unquoted) `@name` already handled multi-word entity names fine at *commit* time (longest-name greedy match), but the *live* mention dropdown (`trailing_mention_query`) only ever looked at the single trailing whitespace-delimited word — so searching a multi-word name mid-typing (`@Jane D...`) never showed a match once the first space was typed, since the query reset. Quoted mentions fix that: `trailing_mention_query` now special-cases an unterminated `@"..."` at the end of the string (checked via "does the last `@\"` have a partner `"` yet") and returns everything after the opening quote — spaces included — as the live search query, closing (returning `None`, handing off to normal `tokenize()`/`parse()`) the instant the closing quote appears. `apply_mention` was changed to take an explicit byte `start` offset instead of re-deriving the replacement boundary from the last whitespace character, since a quoted fragment's start isn't at a whitespace boundary in the usual sense.

Tags gained the same `+"..."`/`-"..."` form for the same reason — bare `+tag` was always a single whitespace-delimited word, so there was previously no way to add a tag containing a space at all. `tokenize()` special-cases `@"`, `+"`, and `-"` before falling into ordinary word-splitting; an *unterminated* quote (still being typed, or just malformed) deliberately falls through to plain word tokenization instead of erroring — same "stays uncolored until it activates" philosophy as everything else in this module. A quoted `@"..."` that doesn't match any real entity also stays plain (quotes and all, visible in the title) rather than silently disappearing.

6 new unit tests (23 total) cover: quoted mention exact-match / no-match, quoted tags with spaces, an unterminated quote staying plain, and the live-query-spans-a-space / closes-on-terminator behavior. Verified via `dx serve` + Playwright: typing `@"Jane D` while a real "Jane Doe" entity exists correctly surfaces it in the dropdown (proving the space-spanning search works, which the bare form can't do), selecting it correctly resolves to the closed `@Jane Doe` form, and `+"corrupt journalist"` submits as a single tag with the space intact, landing in the right entity's file with the right tag — all through the real running UI, no console errors.

## Urgency engine rewrite: wired-up factors + user-editable weights (2026-07-15)

Closes the loop the "advanced fold"/quick-capture work opened: priority/project/scheduled were collectible everywhere but didn't affect ranking at all. New module **`src/urgency.rs`** is now *the* place responsible for computing a moment's priority score — explicitly requested as "a very clear function... so I can articulate why and how things are the way they are." `compute_urgency(moment, all_moments, now, weights) -> UrgencyBreakdown` is the whole story; every other file that touches ranking just calls it.

**Model**: taskwarrior's own "coefficient × indicator" approach — each factor computes an independent 0..1 (or boolean) *indicator* describing how true that factor is, then multiplies by a *weight* that says how much it should matter. This is a deliberate change from the old inline version, which baked scaling directly into each term (e.g. `gravity/100 * 5.0`) — that made the "5.0" essentially invisible and unowned. Now every scaling factor is a named field on `UrgencyWeights`, so "how much do I care about X" is always one editable number, and setting any weight to 0 fully disables that factor with no special-casing needed.

**Factors, each independently weighted**: due date (0..1 ramp, overdue = max), priority H/M/L (flat bonus per level — not indicator×weight like the rest, since the field is already binary per level), has-a-project (flat bonus), scheduled-and-now-active (flat bonus once the scheduled date has passed — a ranking nudge only, does **not** hide the task before then, that enforcement is still a separate deferred item), gravity (the existing -100..100 dial, normalized), age (ramps over 30 days), blocked (penalty, negative by default), blocking (bonus), tags (small bonus, capped at 3). `until` deliberately has no urgency weight at all — it's a lifecycle/expiration field, not an urgency signal, and giving it one would be inventing behavior around a feature (auto-expiry) that doesn't exist yet.

**A real bug caught in the same pass, not a new one introduced**: the original `compute_urgency` (and the `due_at`/`scheduled_at` fields generally) parsed dates via `chrono::DateTime::parse_from_rfc3339` — but `due_at`/`scheduled_at`/`until_at` are actually always written in the bare `"YYYY-MM-DDTHH:MM"` shape a `<input type="datetime-local">` produces (both the Advanced fold and quick-capture write this same shape), which isn't valid RFC3339. So due-date scoring was **silently a no-op for every real due date ever set through the actual UI** — confirmed live via Playwright before assuming the fix worked (a task with `due:today` initially showed no `due` contribution in its tooltip at all). Fixed with a shared `parse_moment_datetime()` that tries RFC3339 first, falls back to the bare datetime-local shape. New regression test (`due_date_in_the_actual_datetime_local_shape_still_scores`) specifically covers this — the pre-fix test suite would have passed even with the bug, since it only tested the RFC3339 shape nothing in the app actually produces for this field.

**Transparency, not just a number**: `UrgencyBreakdown` keeps every factor's contribution named, plus a `.describe()` that renders the nonzero ones largest-first (e.g. `"due +12.0 · priority +6.0 · gravity +0.1"`). The Priority view's score badge carries this as its `title` tooltip — hovering any ranked item's score shows exactly why it's ranked there, live, no separate UI needed.

**Settings UI**: new `UrgencySettingsCmp` (gear icon next to the Priority view header) — a modal listing every weight as a labeled number input, each one applying and persisting immediately on change (no separate Save step, consistent with every other per-field edit in this app), plus "Reset to defaults". Persisted to `localStorage` (`urgency_weights`, JSON-serialized `UrgencyWeights`) the same way `sort_mode`/`active_vault` already are — hydrated in `main.rs`'s existing startup effect, `#[cfg(not(feature = "desktop"))]`-gated like everything else there (desktop has no preference persistence yet, a known pre-existing gap, not solved here).

12 new unit tests in `urgency.rs` (35 total across the whole suite) cover each factor independently, the zero-weight-disables-it property, and the datetime-shape regression above. Verified via `dx serve` + Playwright: created a high-priority/due-today task and a plain task, confirmed correct ranking and the tooltip breakdown, opened Settings, zeroed the due weight, confirmed the ranking/tooltip updated live, reloaded and confirmed the change persisted, then confirmed "Reset to defaults" restored it — no console errors in any run.

## Product vision

What peeplist actually is, and where it's headed — not just this UI pass. Established across sessions, most of it predates this design pass.

**What it is:** a relationship-first task list app — tasks and reminders are organized around the people/entities in the user's life, not just flat todo items. Two core entity concepts beyond generic "tasks":
- **Promises** (`moment_type_id == 2`) are a distinct concept from tasks — a BOLO ("be on the lookout") on something *another person* said they'd do, not an elevated/more-important task type. This exists to counter a specific pain point the user described: they hear a lot of vague commitments ("let's catch up sometime," "I'll get back to you"), have gotten cynical/burned-out not tracking any of it, and default to assuming people are "bullshitting" them. Promises let them actually verify who follows through. Scope: *incoming* commitments from others, not the user's own tasks. Worth periodically re-checking whether the current implementation still matches this intent — it hasn't had a dedicated audit pass.
- **Notes** (`moment_type_id == 3`) — logged observations that were never "planned" or "completable" (e.g. "they brought me flowers"). As of the Distance/Drift work, notes with non-zero `gravity` now meaningfully affect a relationship's computed Distance even without ever being "completed" — see the Distance/Drift entry further down.

**Core product pillars** (the north star for feature/architecture calls, not just this pass):
- Broad compatibility — not locked to one platform/ecosystem.
- End-to-end encryption. **Not implemented at all yet.**
- A native CLI client that connects to the user's account — a first-class client, not an afterthought. **Not implemented at all yet.**
- A developer-facing API for third-party tooling. **Not implemented at all yet.**
- Taskwarrior-style quick-capture input syntax, e.g. `pltask add "pick up groceries" @family pri:h` (`@` for tags, `pri:` for priority). **Not implemented** — the moment composer today is still just a plain title field + entity dropdown, no parsing.

## Build/architecture

- **Frontend-only build (2026-07-14)**: removed the `fullstack` feature from `dioxus` in `Cargo.toml`. It was pulling in `axum`/`tokio`/`dioxus-fullstack`/`dioxus-server` for zero actual use — no `#[server]` functions or `dioxus_fullstack` usage ever existed in this codebase; the app talks to Supabase directly from the client via `reqwest`. Verified: `cargo tree --target wasm32-unknown-unknown -i <those crates>` shows none of them in the actual wasm build graph anymore; `dx serve` restarted clean, app loads with no console errors. (Those package entries may still *appear* in `Cargo.lock` — that's normal, Cargo keeps lock entries reachable via any declared feature combination, not just the active one; it doesn't mean they're compiled in.)
  - **Why**: explicit goal to keep the build divorced from any backend-server mode, so `dx build` can target multiple platforms via Dioxus's own tooling without carrying backend weight. Also mentioned: scaling down VM/hosting needs, and a possible future unikernel for security — deferred, see memory `project_unikernel_idea`, don't act on it unprompted.
  - **Platform targets** (see memory `project_app_vision`): mobile, desktop, cli, web-mobile (reactive), desktop-web — currently focused on **desktop-web and mobile-web**.
  - `Cargo.toml` already had proper `web`/`desktop`/`mobile`/`server` feature gates matching the standard Dioxus template (`default = ["web"]`) — that part was already correct; `fullstack` was the only stray unconditional feature causing the bloat.

- **Deploy script** (`scripts/deploy.sh`, 2026-07-14): the user's actual deploy pain — server has too little CPU/RAM to compile, and M1 Mac binaries don't run on the (different-arch) server. The frontend-only build above sidesteps this: `dx build --platform web --release` now produces *pure static files* (HTML/CSS/JS/WASM), no compiled binary at all, so there's nothing architecture-specific to ship. The script: cleans `target/dx/peeplist/release/web` (so stale content-hashed files from old builds don't linger), runs the release build, sanity-checks `index.html` exists, then `rsync -avz --delete`s the output to a server over SSH. Deploy target is config, not hardcoded — `DEPLOY_HOST`/`DEPLOY_PATH` (+ optional `DEPLOY_USER`/`DEPLOY_PORT`) via env vars or `scripts/.env.deploy` (gitignored; `scripts/.env.deploy.example` is the template). `BUILD_ONLY=1` skips the rsync leg entirely. Verified end-to-end except the actual rsync (no real server to test against): build-only mode produces a working `index.html` + `assets/`, and the missing-config path fails loudly with exit code 1 and a clear message rather than silently no-opping.
  - **Also fixed while building this**: `dx build --release` was crashing at the `wasm-opt` step (`SIGABRT`, "compile unit size was incorrect... UNREACHABLE executed at DWARFEmitter.cpp") — this binaryen version chokes on DWARF debug info, which Dioxus generates by default (`--debug-symbols true`) even in release builds. The script passes `--debug-symbols false`, which both fixes the crash and produces a smaller optimized binary (1.6MB vs. the 3.2MB *unoptimized* output you got when wasm-opt failed and was silently skipped). Not something the user needs debug symbols for in a production deploy anyway.
