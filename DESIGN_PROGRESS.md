## Product vision

What peeplist actually is, and where it's headed — not just this UI pass. Established across sessions, most of it predates this design pass.

**What it is:** a relationship-first task list app — tasks and reminders are organized around the people/entities in the user's life, not just flat todo items. Two core entity concepts beyond generic "tasks":
- **Promises** (`moment_type_id == 2`) are a distinct concept from tasks — a BOLO ("be on the lookout") on something *another person* said they'd do, not an elevated/more-important task type. This exists to counter a specific pain point the user described: they hear a lot of vague commitments ("let's catch up sometime," "I'll get back to you"), have gotten cynical/burned-out not tracking any of it, and default to assuming people are "bullshitting" them. Promises let them actually verify who follows through. Scope: *incoming* commitments from others, not the user's own tasks. Worth periodically re-checking whether the current implementation still matches this intent — it hasn't had a dedicated audit pass.
- **Notes** (`moment_type_id == 3`) — logged observations that were never "planned" or "completable" (e.g. "they brought me flowers"). As of the Distance/Drift work, notes with non-zero `gravity` now meaningfully affect a relationship's computed Distance even without ever being "completed" — see the Distance/Drift entry further down.

**Core product pillars** (the north star for feature/architecture calls, not just this pass):
- Broad compatibility — not locked to one platform/ecosystem.
- End-to-end encryption. **Not implemented at all yet.**
- A native CLI client that connects to the user's account — a first-class client, not an afterthought. **Not implemented at all yet.**
- A developer-facing API for third-party tooling. **Not implemented at all yet.**
- Taskwarrior-style quick-capture input syntax, e.g. `pltask add "pick up groceries" @family pri:h` (`@` for tags, `pri:` for priority). **Not implemented** — the moment composer today is still just a plain title field + entity dropdown, no parsing.
- A "graph/orbit view" — a visual map of relationships. **Shipped, first pass** (sidebar "Graph View", `src/components/graph.rs`) — you at center, entities positioned by Distance. `src/todo.md`'s original spec for this ("Graph Features") goes further than what's built: **three switchable view modes** (overview / gravity / reaction — only one generic mode exists today), **timeline scrubbing** through graph history (not built), tap-a-node detail (done — click opens the entity). "Drift velocity per entity" from that spec is essentially `entities.drift`, which exists but isn't visualized on the graph itself yet (only in Stats/Info).

**Two things from `src/todo.md` worth flagging explicitly, found while writing this section:**
- **"Urgency Engine"** in `todo.md` ("urgency calculated from due date, gravity, drift velocity, and promise age... blocked by dependencies... blocking other moments... ranked queue") is essentially the spec for what got built as **Priority view** (`compute_urgency` in `src/components/moment.rs`) — due date, gravity, age, blocked/blocking are all there. **Gap**: the original spec says "drift velocity," i.e. incorporate `entities.drift`/computed Distance into urgency — the current formula doesn't use Distance/Drift at all (Distance/Drift didn't exist yet when Priority was built). Worth revisiting once Priority engine tuning comes up.
- **"Crisis View"** in `todo.md` is a *different* feature than what exists today under that name might suggest — the original spec is a dashboard ("closest entities at a glance, recent moments logged by others for them, recent moments done for others, open promises made to them, evidence of recent positive reactions"), not a graph. The sidebar used to have a "Crisis View" nav link, but the user explicitly asked for it to be **renamed to "Graph View" and repurposed** for the orbit visualization — so the original Crisis-View-the-dashboard concept is currently unbuilt *and* has no nav slot pointing at it anymore. If it's ever wanted, it needs a new home in the UI, not a revival of the old link.
- Also unbuilt, no session has touched these: **Account/Settings** (delete account + data export, notification/reminder prefs, default-entity-to-self setting), **Social/Sharing** (share a moment, invite someone), **Paid Tier** (entity-count limits, feature gating for graph history/urgency engine/crisis view — implies a monetization model not decided anywhere yet).

**Concrete platform-target decision** (2026-07-14, see "Build/architecture" below for the technical half of this): the app is being kept frontend-only specifically so `dx build` can target multiple platforms via Dioxus's own tooling without dragging backend weight along. Named targets: **mobile, desktop, cli, web-mobile (reactive), desktop-web** — current focus is **desktop-web and mobile-web**. The native-CLI pillar above should eventually be one of these targets, not a separate codebase.

**`src/todo.md`** (tracked, not gitignored — this is the user's own product notes, not an AI-tooling doc, leave it alone) is the fuller, original backlog in the user's own words — read it when picking "what's next," it covers things no session has discussed yet (see the four bullets above).

- `src/views/auth.rs`: login previously failed **silently** (error only went to `clog!`/browser console). Now shows a visible error message and redirects to `Route::Home` automatically if the user is already authenticated (`state.auth_token` is `Some`).

Fixed:
  - `src/api/client.rs`: added `auth_get`, an `/auth/v1/{path}` request authenticated with the *user's* token (existing `auth_post` always used the anon key, which is right for login but wrong for validating a session).
  - `src/api/auth.rs`: added `get_current_user` (hits `GET /auth/v1/user` to check a token is still accepted) and `refresh_access_token` (calls Supabase's `grant_type=refresh_token` flow).

  - `src/layouts/navbar.rs` (`Navbar`, since it's the layout mounted inside the Router and can call `navigator()`): on every load, validates the cached token via `get_current_user`; if rejected, clears storage + `AppState` and redirects to `Route::LoginCMP`. Also starts a background loop (`TOKEN_REFRESH_INTERVAL_MS` = 50 min, via `gloo_timers`) that proactively calls `refresh_access_token` so a live session ideally never reaches the dead-token state at all; if refresh itself fails (e.g. refresh token revoked), it logs the user out the same way.
  - Verified via Playwright: injecting a fake token into `localStorage` and reloading now correctly logs the "Session check failed..." message, clears storage, and lands on `/login`; a normal unauthenticated load (no token) is unaffected.

## Session — 2026-07-12: bug fixes, activity bar rebuild, and first real features

Long session, several sub-parts. Everything below is **already committed** (`784a011 "so many changes its almost not worth listing"`) — see "State of the working tree" for what that leaves outstanding.

### Bug fixes
- **RefCell `AlreadyBorrowed` panic on logout** (`src/layouts/navbar.rs`, `src/components/entity.rs`, `src/components/moment.rs`, `src/views/home.rs`): the pattern `match some_async_fn(token.read().clone()...).await { ... }` keeps the `token.read()` guard alive for the *entire match block* (Rust's temporary-lifetime-extension rule for match scrutinees), including across the `.await` and through the `Ok`/`Err` arm. With ~11 call sites doing this on the shared `auth_token` signal, any concurrent write (e.g. the session-check logout in `Navbar`) could hit an already-borrowed signal and panic. ### Ordering, tags, dependencies, priority — no schema migration needed
Discovered the `moments` table already had two unused columns: `metadata jsonb` and `depends_on bigint`. Built all of the below on top of them rather than asking for a migration:
- **Stable order** (`src/api/moment.rs`, `getMoments`): added `&order=id.asc` — previously had no `order=` param at all, so Postgres/PostgREST returned rows in unspecified (effectively random-looking) order on every refresh.
- **Tags**: `MomentMetadata { tags: Vec<String>, sort_index: Option<f64> }` stored in `metadata`. Add/remove chips in `ab_task_cmp`. Sidebar (`src/components/sidebar.rs`, `tag_list_cmp`) shows the distinct set of all tags in use, click to filter (`state.tag_filter`, applied in `home.rs` alongside entity filtering).
- **Sort modes** (`MomentListCmp`): Default (id order) / Due date / Custom (drag-and-drop, HTML5 drag events, persists position to `metadata.sort_index`). Chosen mode persisted to `localStorage` (`sort_mode` key) so it survives reloads — `SortMode` enum + `as_storage_str`/`from_storage_str` in `src/main.rs`.
- **Depends on**: `moments.depends_on` now exposed and wired — a picker in `ab_task_cmp` (scoped to other open moments in the same entity), a "Blocked by …" indicator, and a reverse "Blocking N other open moment(s)" line. **Caveat**: this is a single dependency (`bigint`, not a join table) — real taskwarrior-style multi-dependency would need a `moment_dependencies` table (see memory `project_db_coordination`).
- **Priority view** (`PriorityViewCmp`, `src/components/moment.rs`, new `View::Priority` reachable via a sidebar nav link): all open tasks/promises across every entity, ranked by a taskwarrior-esque urgency score — `compute_urgency()` combines due-date proximity (ramps to +12 as due/overdue), gravity (scaled to ±5), a small age bonus (caps at +2 after 30 days), a heavy blocked penalty (-8), and a blocking bonus (+8). Coefficients are a first draft (commented inline), not tuned against real usage yet.
- **Due date shown in the center list** (`MomentCmp`): small trailing label (`"%b %d"`), red/bold if overdue and not completed.

## Build/architecture

It was pulling in `axum`/`tokio`/`dioxus-fullstack`/`dioxus-server` for zero actual use — no `#[server]` functions or `dioxus_fullstack` usage ever existed in this codebase; the app talks to Supabase directly from the client via `reqwest`.
